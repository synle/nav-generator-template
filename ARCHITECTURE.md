# nav-generator-template — Architecture

## High-Level Overview

`nav-generator-template` is a GitHub-template repository that downstream users fork to publish their own personal bookmark/navigation page on GitHub Pages. The runtime engine lives in the upstream library [`synle/nav-generator`](https://github.com/synle/nav-generator); this repo contains only the per-user *schema* (the list of links) plus a thin HTML/JS shell that loads the engine from `synle.github.io/nav-generator`.

Conceptually:

- The user defines a plain-text schema (sections + `label | url` pairs) inside `index.js`.
- A `NavBeforeLoad` DOM event handler hands that schema to the upstream engine via `renderSchema(...)`.
- The engine (loaded by `index.html` as a remote `<script>`) parses the schema and renders the navigation UI.
- A service worker hosted on the upstream site is registered for offline support.
- GitHub Actions builds the site (refreshing the HTML shell from the upstream template) and deploys it to GitHub Pages.

There is no application code beyond the schema and shell — the template's only job is to be a minimal, forkable starting point.

## Key Directories

- `/` (root) — Entire site. Static, no nesting. `index.html` is served as-is; there is no `src/` or `dist/`.
- `.github/workflows/` — Single CI workflow that delegates to the shared `synle/workflows` build-and-deploy pipeline.

The repo is intentionally flat: a template has to stay trivial for downstream forks.

## Important Files

- `index.js` — **The only file end users edit.** Listens for `NavBeforeLoad`, defines `SITE_SCHEMA` (multiline string: `! Title`, `@ Emoji`, `# section`, `label | url`), and calls `renderSchema(SITE_SCHEMA)` to hand the schema to the upstream engine.
- `index.html` — HTML shell. Includes `index.js`, then loads the upstream engine `https://synle.github.io/nav-generator/index.js?hasCustomNavBeforeLoad=1`, then registers the upstream service worker `https://synle.github.io/nav-generator/sw-nav.js`. Auto-checks for SW updates hourly and reloads on `controllerchange`. **This file is regenerated by `build.sh`** — manual edits are overwritten on every build.
- `build.sh` — Two-line bootstrap. Pulls the canonical `index.template.html` and `dev.sh` from `synle.github.io/nav-generator` via `curl`, overwriting the local copies. This is how the template stays in sync with the upstream engine without each fork having to redownload manually.
- `dev.sh` — Local dev loop. Streams `synle/workflows`'s shared `dev.sh` via `curl … | bash`, watching `*.json *.scss *.jsx *.js` and running `npm run start`.
- `package.json` — Declares `prettier` as the sole devDependency. Scripts: `build` (= `bash build.sh`), `dev` (= `bash dev.sh`), `start` (= `npx http-server --cors -a 127.0.0.1 .`), `format`, `clean`.
- `package-lock.json` — Lockfile for the prettier devDep.
- `.github/workflows/build-main.yml` — CI entrypoint. On `push`/`pull_request` to `master`/`main`, delegates to the reusable workflow `synle/workflows/.github/workflows/build-and-commit-sh.yml@main` with `deploy-to-pages: true`. Grants `contents: write`, `pages: write`, `id-token: write`. Uses a `concurrency` group keyed by workflow + ref with `cancel-in-progress: true`.
- `README.md` — User-facing instructions for forking the template, enabling GitHub Pages, and accessing the live site.
- `DEV.md`, `CLAUDE.md` — Developer/agent guidance. `CLAUDE.md` notably enforces squash-merge policy and prescribes `raw.githubusercontent.com` for raw GitHub fetches.
- `.gitignore`, `.gitattributes`, `.prettierignore` — Standard hygiene; nothing custom of architectural note.

## Build & Release Flow

1. **Local dev**
   - `npm run dev` — runs `bash dev.sh`, which pipes the shared `dev.sh` from `synle/workflows` through `bash`. That harness watches the listed glob patterns and re-runs `npm run start` (an `http-server` on `127.0.0.1:8080` with CORS).
   - `npm run build` — runs `build.sh`, which re-fetches the upstream `index.template.html` and `dev.sh` so the local shell stays current with the engine.
   - `npm run format` — runs prettier across the repo with a 140-column print width.

2. **CI build & deploy** (`.github/workflows/build-main.yml`)
   - Triggers on push and PR against `master` or `main`.
   - Delegates entirely to the reusable workflow `synle/workflows/.github/workflows/build-and-commit-sh.yml@main`.
   - That reusable workflow runs `build.sh` (refreshing `index.html` / `dev.sh` from upstream), commits any drift back to the branch, and — because `deploy-to-pages: true` — publishes the workspace to GitHub Pages using the `pages: write` + `id-token: write` permissions granted here.
   - `concurrency` cancels in-progress runs on the same ref so a fast follow-up push supersedes its predecessor.

3. **Release**
   - There is no tagged release artifact. The site itself *is* the release: a successful `main` build deploys to `https://<user>.github.io/nav-generator-template/` (or the fork's equivalent).
   - The engine and service worker are pinned by URL to `synle.github.io/nav-generator`, so upstream engine changes flow to every deployed instance on the next page load / SW update cycle without re-deploying the template.
